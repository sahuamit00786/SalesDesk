Auth & Access

cross-cutting · sequential

The one flow every other flow sits on top of. Worth testing in isolation before anything else, since a break here blocks every other module.

  1. RegisterCompany + first user created
  2. Verify emailOTP, resend available
  3. Log inAccess + refresh token issued
  4. Accept inviteAlt path, if invited instead of self-registered
  5. OnboardingFirst-login company/workspace setup
  6. Workspace selectedx-workspace-id now sent on every request
  7. DashboardLanding screen

Separate alt path: forgot-password → OTP → reset-password, independent of the flow above. Permission keys follow the same grouping as the sidebar sections below (main.*, engage.*, manage.*, automate.*, settings.*) — a role that's missing a key can't see that section at all, worth testing per role tier.

Leads

The longest-lived record in the system — everything else eventually points back to a lead.

  1. Capture — manual, CSV import, web form, or campaign staging
  2. Assign — manually, or round-robin distribution
  3. Work — notes, tasks, follow-ups, logged calls/emails
  4. Qualify — status moves through configurable categories
  5. Convert — promoted to Opportunity or straight to Deal

Automate

/automation · /campaigns · /forms

Automation (workflows)

No-code rule builder — DAG canvas of triggers, conditions, delays, actions.

  1. Define trigger — lead/email event
  2. Build the graph — conditions, delays, actions
  3. Test run — dry run before going live
  4. Publish — version becomes active
  5. Runs — every firing logged with step-by-step history
/automation/automation/:idautomate.workflows

Campaigns

Multi-step outreach to a lead segment, tracked as its own mini-funnel.

  1. Create — pick leads, team, assignment rule
  2. Stage leads into the campaign funnel
  3. Move leads across stages — each move logged to stage history
  4. Review the campaign report
/campaigns/new/campaigns/:id/campaigns/:id/reportautomate.campaigns

Web forms & lead capture

The outside-in path into Leads — no login required for the visitor.

  1. Build — drag-drop fields in the form builder
  2. Publish & embed — snippet placed on an external site
  3. Visitor submits — public endpoint, no auth
  4. Lead auto-created — source tagged as the form

Known gaps

read before you file a bug

HR (Attendance & Leave) — UI disabled, API live

The HR nav section and its routes are commented out in the current build — you won't find Attendance or Leave in the sidebar, and that's expected, not a bug. The backend is fully implemented and reachable directly (/leave/*, /attendance), and the Leave and Employee-Monthly tabs still exist under Reports — expect those tabs to show empty rather than error, since nothing is writing HR data while the UI path is off.

Email sequences & lead scoring rules — API exists, no dedicated page found

Both have full CRUD endpoints (/email-sequences, /scoring-rules) but neither turned up as a top-level sidebar page in this pass. If you find them surfaced inside another screen (Lead configuration and Templates are the likely spots), note where — otherwise treat as backend-only for now.
